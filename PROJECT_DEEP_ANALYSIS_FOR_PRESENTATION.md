# STARLOG 프로젝트 정밀 분석 보고서

## 1. 문서 목적
이 문서는 STARLOG 프로젝트를 발표할 때 바로 사용할 수 있도록 작성된 정밀 분석 자료다.
분석은 코드, 설정, 템플릿, 테스트, 운영 문서를 함께 검토하는 방식으로 진행했다.

## 2. 프로젝트 한 줄 정의
STARLOG는 Spring Boot 기반의 학교 커뮤니티 플랫폼으로, 인증/게시글/댓글/추천/이미지 업로드/한글 고급 검색을 핵심 기능으로 제공한다.

## 3. 분석 범위와 기준

### 3.1 분석 범위
- 백엔드: Controller, Service, Repository, Entity, Exception, Security 설정
- 프론트엔드: Thymeleaf 템플릿, 주요 JS 동작(검색, 실시간 폴링)
- 설정: dev/prod 프로필, DB, 업로드 경로, 세션/쿠키 정책
- 품질: 테스트 코드 존재 여부와 실행 결과
- 운영: systemd, Cloudflare Tunnel, 배포/백업 가이드

### 3.2 분석 기준
- 기능 완성도
- 보안성
- 성능/확장성
- 운영 안정성
- 문서와 코드의 일치성

## 4. 기술 스택 요약

| 구분 | 내용 |
| --- | --- |
| 언어/런타임 | Java 17, Spring Boot 3.5.7 |
| 웹 계층 | Spring MVC + Thymeleaf |
| 데이터 계층 | Spring Data JPA (Hibernate 6), H2(dev), MySQL(prod) |
| 인증/보안 | Spring Security, 세션 인증, BCrypt, CSP/HSTS/Referrer Policy |
| 프론트 동작 | Vanilla JS, GSAP(문서 기준), 실시간 폴링 |
| 한글 검색 | Java HangulUtils + 브라우저 es-hangul(getChoseong) |
| 배포/운영 | Maven Wrapper, systemd, Cloudflare Tunnel |

## 5. 아키텍처 분석

### 5.1 구조 개요
전형적인 계층형 아키텍처다.
- Presentation: Controller + Thymeleaf Template + Static JS/CSS
- Business: Service
- Persistence: Repository(JPA)
- Domain: Entity

### 5.2 장점
- 책임 분리(Controller/Service/Repository)가 명확하다.
- 실서비스 기능 중심으로 코드가 빠르게 구현되어 있다.
- 보안 관련 방어 로직(입력 검증, XSS 필터링, 브루트포스 제한)이 실제 코드에 반영되어 있다.

### 5.3 구조적 한계
- DTO 사용이 API 일부에만 적용되어 있고, 일부 엔드포인트는 엔티티를 직접 반환한다.
- 게시글 고급 검색이 메모리 필터 기반이라 데이터 증가 시 확장성 리스크가 크다.
- 운영 문서 대비 자동화(헬스체크, 관측 지표, CI 기반 품질 게이트) 근거가 부족하다.

## 6. 핵심 기능 정밀 분석

### 6.1 인증/회원
- 회원가입: username/password/email 유효성 검증 + 중복 검사
- 로그인: 세션 기반 인증
- 보안: 로그인 실패 누적 5회 시 15분 잠금(DB 영속)
- 로그아웃: POST만 허용, 세션 무효화
- 회원탈퇴: 비밀번호 재확인 후 연관 데이터 삭제

평가:
- 보안적 기본기는 충실하다.
- 계정 잠금이 DB에 남아 서버 재기동 후에도 유지되는 점이 실무적으로 우수하다.

### 6.2 게시글/댓글/추천
- 게시글 CRUD, 권한 검증(작성자 확인), 조회수 증가
- 댓글 + 대댓글(부모 댓글의 동일 게시글 소속 검증)
- 좋아요/싫어요 토글(동일 클릭 취소, 반대 클릭 전환)

평가:
- 커뮤니티 기본 기능은 충분히 완성됐다.
- 댓글/추천의 서버측 무결성 검증이 잘 들어가 있다.

### 6.3 파일 업로드
- 확장자/MIME/크기 검증
- UUID 파일명
- 업로드 경로 분리
- 회원탈퇴 시 프로필 파일 정리 시 canonical path 검증

평가:
- 업로드 보안 체크 포인트를 폭넓게 반영했다.
- 게시글 이미지 물리 파일 정리 정책은 문서화/자동화가 더 필요하다.

### 6.4 한글 검색
- 기본 검색: DB Like 기반
- 고급 검색: 초성, 자모 분리, 영타 변환 지원
- 점수 기반 정렬 로직 존재

평가:
- 사용자 체감 품질이 좋은 검색 UX를 제공한다.
- 현재 구현은 전체 게시글를 메모리로 읽어 필터링하므로 대용량에서 병목 가능성이 높다.

### 6.5 실시간 UX
- 새 글: 5초 폴링
- 인기글: 30초 폴링
- 목록 화면에서 즉시 반영

평가:
- 구현 난이도 대비 체감 가치가 높은 방식이다.
- 폴링 기반이므로 트래픽 증가 구간에서 API 부하 관리가 필요하다.

## 7. 데이터 모델 분석

핵심 엔티티:
- User
- Post
- PostImage
- Comment(자가 참조 parent/children)
- Lh(Like/Hate)
- LoginAttemptEntity

관계 요약:
- User 1:N Post
- Post 1:N PostImage
- Post 1:N Comment
- Post 1:N Lh
- Comment 1:N Comment(대댓글)

평가:
- 커뮤니티 도메인에 필요한 관계가 잘 표현됐다.
- User.email은 애플리케이션 레벨 중복 검사만 존재하며 DB unique 제약이 없어 동시성 시나리오 리스크가 있다.

## 8. 보안 분석

### 8.1 강점
- BCrypt 비밀번호 해시
- 입력 길이 및 형식 검증
- Jsoup Safelist.none()로 HTML 제거
- 세션 고정 공격 방지(로그인 시 세션 재발급)
- 보안 헤더(CSP, HSTS, Referrer-Policy)
- 업로드 파일 검증

### 8.2 보완 필요 지점
1. Spring Security는 anyRequest().permitAll() 구성이라 인증/인가 대부분이 컨트롤러 수동 로직에 의존한다.
2. CSP에 unsafe-inline이 포함되어 있어 XSS 방어 강도를 더 높일 여지가 있다.
3. 공개 API 응답에서 엔티티 직접 반환 비중이 있어, 향후 필드 노출 회귀 위험이 있다.

## 9. 성능 및 확장성 분석

### 9.1 현재 강점
- 페이지네이션 적용
- 정적/업로드 리소스 캐시(30일)
- 프로필 분리(dev/prod)

### 9.2 잠재 병목
1. 고급 검색이 전체 게시글 메모리 스캔 방식
2. 새 글/인기글 폴링의 누적 요청량 증가
3. Open Session In View 경고 상태(기본값 활성)

### 9.3 확장 방향
- 고급 검색을 DB 인덱스 + 전용 검색 컬럼(초성/자모 전처리)로 이동
- 빈번 조회 API 캐시(짧은 TTL) 도입
- OSIV 비활성화 검토 + fetch 전략 최적화

## 10. 테스트와 품질 현황

### 10.1 테스트 상태
- 단위 테스트 존재: UserService, PostService, HangulUtils
- 스모크 테스트 존재: contextLoads
- Maven 테스트 실행 확인: 실패 없이 완료

### 10.2 품질 관점 평가
- 핵심 서비스 로직의 기본 테스트는 갖췄다.
- 컨트롤러 통합 테스트, 보안 회귀 테스트, 파일 업로드 엣지케이스 테스트는 보강 필요.

## 11. 문서-코드 정합성 점검

다음 항목은 발표 시 반드시 선제적으로 설명하면 신뢰도가 올라간다.

1. 운영 문서에 언급된 일부 보안/제한 사항은 코드에서 직접 근거를 찾기 어려운 항목이 있다.
2. 문서에서 강조하는 API Rate Limit은 현재 코드 기준 명시적 서버 구현 근거가 부족하다.
3. README 일부 설명(H2 인메모리)은 현재 dev 설정(file 기반 H2)과 표현 차이가 있다.

## 12. 종합 평가

기능 완성도와 실사용 관점의 현실적 구현력이 높은 프로젝트다. 특히 커뮤니티 서비스 핵심 동선(인증, 글/댓글, 검색, 추천, 마이페이지)을 빠짐없이 구현했고, 보안 방어도 기초 이상 수준으로 반영했다.

- 과제: 확장성, 보안 정책의 중앙화, 문서-코드 완전 동기화

## 13. 우선순위 개선 로드맵

### 13.1 단기(1~2주)
1. 문서-코드 불일치 항목 정리 및 수정
2. 엔티티 직접 반환 API를 DTO 기반으로 통일
3. 이메일 DB unique 제약 추가 + 마이그레이션 적용
4. 고급 검색 성능 측정 기준 수립(데이터 1만/10만 기준)

### 13.2 중기(3~6주)
1. 검색 인덱싱 구조 도입
2. 인기글/새글 API 캐시 적용
3. 통합 테스트와 보안 회귀 테스트 확대
4. 관측성 지표(요청량, 오류율, 응답시간) 대시보드 구축

### 13.3 장기(6주+)
1. 이벤트 기반 실시간 갱신(WebSocket/SSE) 검토
2. 인증/인가를 Spring Security 중심 정책으로 재정렬
3. 운영 자동화(CI 품질 게이트, 배포 검증 체크리스트 자동화)

## 14. 발표용 스크립트(10분 기준)

### 14.1 0~1분: 문제 정의
- 학교 커뮤니티에서 필요한 것은 빠른 글 공유, 쉬운 검색, 신뢰 가능한 운영이다.
- STARLOG는 이 세 가지를 Spring Boot 기반으로 통합 제공한다.

### 14.2 1~3분: 아키텍처
- Controller-Service-Repository 계층형 구조
- 세션 인증, JPA 기반 데이터 처리, Thymeleaf 서버 렌더링
- dev/prod 프로필 분리와 systemd 운영 방식

### 14.3 3~6분: 핵심 기능 데모 포인트
- 회원가입/로그인/로그아웃/탈퇴
- 게시글 작성-수정-삭제와 이미지 업로드
- 댓글/대댓글, 추천/비추천
- 고급 한글 검색(초성, 자모, 영타)

### 14.4 6~8분: 보안/운영 포인트
- 브루트포스 방어(DB 영속)
- XSS/업로드 보안 검증
- 보안 헤더, 쿠키 정책
- Cloudflare Tunnel + systemd 운영

### 14.5 8~10분: 리스크와 개선 계획
- 검색 확장성, API 응답 표준화, 문서-코드 일치성
- 단기/중기/장기 로드맵으로 단계적 고도화 계획 제시

## 15. 예상 Q&A

Q1. 왜 고급 검색을 메모리 방식으로 시작했는가?
A1. 사용자 체감 품질을 빠르게 확보하기 위한 선택이었다. 현재는 데이터 증가를 전제로 인덱싱 기반으로 전환할 계획이다.

Q2. 보안은 Spring Security로 완전히 통제되는가?
A2. 현재는 세션 인증과 컨트롤러 권한 검증을 병행하는 구조다. 다음 단계에서 정책 중심 구조로 정리할 예정이다.

Q3. 운영 안정성은 어떻게 확보했는가?
A3. systemd 상시 기동, Cloudflare Tunnel HTTPS, DB 백업 가이드를 갖추고 있다. 다음 단계는 모니터링 지표 자동화다.

Q4. 테스트는 충분한가?
A4. 핵심 서비스 단위 테스트는 존재하고 실행도 정상이다. 다만 통합/보안 회귀 테스트를 확대할 필요가 있다.

## 16. 결론
STARLOG는 기능 구현력과 실사용 관점의 완성도가 높은 프로젝트다. 지금 시점의 최적 전략은 기능 추가보다 품질 고도화다. 즉, 검색 확장성, 보안 정책 중앙화, 테스트 및 관측성 강화를 통해 운영 신뢰도를 단계적으로 끌어올리는 것이 가장 큰 성과를 만든다.
