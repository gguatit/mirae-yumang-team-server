## Java / Spring Boot

**1. Java 17을 사용한 이유는 무엇인가요? Java 8이나 11과 어떤 차이가 있나요?**

Spring Boot 3.x부터 Java 17을 최소 요구 버전으로 올렸습니다. Spring Boot 3은 이전 버전과 달리 javax 패키지 대신 jakarta 패키지를 사용하고, GraalVM 네이티브 이미지를 공식 지원하는 등 많은 변화가 있었습니다. Java 17은 LTS 버전이라 장기 지원이 보장되고, record 타입, sealed class, switch 표현식 등 생산성을 높여주는 기능들이 추가되어 선택했습니다.

**2. @Service, @Controller, @Configuration 어노테이션의 역할을 각각 설명해주세요.**

@Service는 비즈니스 로직을 처리하는 서비스 계층임을 나타냅니다. @Controller는 HTTP 요청을 받고 응답을 반환하는 웹 컨트롤러임을 나타냅니다. @Configuration은 스프링 빈을 정의하는 설정 클래스임을 나타냅니다. 세 가지 모두 @Component의 구체화된 버전으로, 스프링이 컴포넌트 스캔을 통해 자동으로 빈으로 등록하게 해줍니다.

**3. @Autowired 대신 생성자 주입을 사용한 이유는 무엇인가요?**

생성자 주입이 권장되는 방식입니다. 그 이유는: 첫째, 필드를 final로 선언할 수 있어 불변성이 보장됩니다. 둘째, 의존성이 누락되면 컴파일 타임에 객체 생성이 실패해서 문제를 빨리 발견할 수 있습니다. 셋째, 순환 참조를 감지할 수 있습니다. @Autowired 필드 주입은 리플렉션을 사용해서 런타임에 주입되므로 final을 사용할 수 없고, 테스트하기 어렵습니다.

**4. Spring Boot에 톰캣이 내장되어 있다는데, jar 파일 하나로 실행되는 원리가 무엇인가요?**

Spring Boot Maven/Gradle 플러그인이 빌드 시점에 내장 톰캣 라이브러리와 애플리케이션 클래스를 하나의 jar로 패키징합니다. 이 jar의 MANIFEST.MF에 Main-Class가 설정되어 있고, spring-boot-loader가 내장 톰캣을 시작한 다음 우리 애플리케이션의 main 메서드를 호출합니다. 외부 톰캣 설치 없이 java -jar 명령어 하나로 실행됩니다.

**5. RestTemplate을 사용했는데, WebClient나 Feign Client 같은 대안들과 비교했을 때 왜 RestTemplate을 선택했나요?**

이 프로젝트에서 NEIS API 호출은 단순한 GET 요청 몇 개로 충분했기 때문에, 가장 익숙하고 간단한 RestTemplate을 선택했습니다. WebClient는 논블로킹/리액티브 방식이라 복잡도가 높고, Feign Client는 선언적 클라이언트지만 추가 의존성이 필요합니다. 간단한 HTTP 클라이언트 호출에는 RestTemplate이 가장 실용적이었습니다.

**6. @ControllerAdvice나 @ExceptionHandler 같은 전역 예외 처리는 사용하지 않은 이유가 있나요?**

현재 API 호출 실패 시 서비스 레벨에서 try-catch로 처리하고 빈 리스트를 반환해서, 템플릿에서 '데이터 없음'으로 표시되도록 하고 있습니다. 전역 예외 처리는 규모가 더 커지거나 다양한 예외 케이스가 생기면 도입할 계획입니다. 현재는 각 예외 상황을 개별 처리해도 관리할 만한 수준입니다.

**7. 500 에러가 발생하면 사용자에게 어떤 화면이 보이나요?**

Spring Boot는 기본적으로 /error 엔드포인트를 제공하고, BasicErrorController가 white-label 에러 페이지를 렌더링합니다. 별도로 error.html을 만들거나 ErrorController를 커스텀하지 않아서 현재는 스프링 부트 기본 에러 페이지가 표시됩니다. 추후 404, 500 등 상태 코드별로 커스텀 에러 페이지를 추가할 계획입니다.

**8. 스프링 부트의 auto-configuration이 어떻게 동작하는지 설명해주세요.**

spring-boot-autoconfigure 라이브러리의 META-INF/spring.factories 또는 org.springframework.boot.autoconfigure.AutoConfiguration.imports 파일에 등록된 수많은 설정 클래스들이 @ConditionalOnClass, @ConditionalOnMissingBean 같은 조건부 어노테이션을 기반으로 동작합니다. 예를 들어 spring-webmvc 의존성이 있으면 자동으로 DispatcherServlet을 설정하고, Thymeleaf 의존성이 있으면 ThymeleafViewResolver를 자동으로 등록합니다.

**9. application.properties에 API 키를 저장했는데, 실제 프로덕션 환경에서는 어떻게 관리해야 하나요?**

운영 환경에서는 다음과 같은 방법을 사용합니다: 1) OS 환경변수로 설정하고 application.properties에서 ${NEIS_API_KEY} 형태로 참조. 2) 외부 설정 서버(Spring Cloud Config) 사용. 3) Docker/Kubernetes의 Secret 리소스 사용. 현재는 개발 편의를 위해 properties에 평문으로 저장했지만, 실제 배포 환경에서는 환경변수로 주입하는 방식으로 전환할 예정입니다.

**10. Bean의 생명주기에 대해 설명해주세요.**

스프링 빈의 생명주기는: 1) 객체 생성(생성자 호출) 2) 의존성 주입 3) InitializingBean의 afterPropertiesSet() 또는 @PostConstruct 메서드 실행 4) 빈 사용 5) 소멸 단계에서 DisposableBean의 destroy() 또는 @PreDestroy 메서드 실행 순서로 진행됩니다. 우리 프로젝트에서는 특별히 라이프사이클 콜백을 구현하지 않고 스프링 기본 동작에 맡기고 있습니다.

## MVC 패턴 / 아키텍처

**11. Spring MVC에서 Model, View, Controller의 역할을 각각 설명해주세요. 왜 이렇게 나누는 걸까요?**

Controller는 HTTP 요청을 받아 파라미터를 검증하고 적절한 Service를 호출합니다. Model은 화면에 표시할 데이터를 담는 컨테이너입니다. View(Thymeleaf 템플릿)는 Model 데이터를 받아 HTML을 생성해서 응답합니다. 이렇게 나누면 관심사가 분리되어 유지보수와 테스트가 쉬워집니다. 예를 들어 우리 프로젝트에서 SchoolController는 요청을 받아 Service를 호출하고, 그 결과를 Model에 담아 school.html로 전달합니다.

**12. Controller - Service - Config 레이어로 나눈 이유는 무엇인가요? 각 레이어가 하는 일을 설명해주세요.**

Config: RestTemplate, 캐시 등 설정과 Bean 생성을 담당합니다. Service: NEIS API 호출, 데이터 가공, 캐싱 등 비즈니스 로직을 처리합니다. Controller: HTTP 요청/응답을 처리하고, Service 호출 결과를 Model에 담아 View로 전달합니다. 이렇게 나누면 각 레이어의 책임이 명확해져서, 예를 들어 API 연동 방식이 바뀌어도 Controller 코드는 변경할 필요가 없습니다.

**13. 계층형 아키텍처의 장단점은 무엇인가요?**

장점: 관심사 분리로 코드 이해와 유지보수가 쉽고, 각 계층을 독립적으로 테스트할 수 있으며, 계층별로 개발자가 분업하기 좋습니다. 단점: 계층이 많아지면 단순한 기능 변경에도 여러 파일을 수정해야 할 수 있고, 불필요한 추상화가 과해질 위험이 있습니다. 작은 프로젝트에서는 오히려 복잡도를 높일 수 있습니다.

**14. DTO(Data Transfer Object)가 무엇이고, 왜 사용하나요?**

DTO는 계층 간 데이터 전송을 위한 단순한 객체입니다. NEIS API의 JSON 응답을 Java 객체로 매핑할 때 사용합니다. 예를 들어 MealServiceResponse, NeisTimetableRow 같은 DTO 클래스가 있습니다. DTO를 사용하면 API 응답 구조 변경에 유연하게 대처할 수 있고, 엔티티를 외부에 직접 노출하지 않아 보안상 좋습니다.

**15. URL 파라미터(?grade=&classNum=)로 학년/반을 받는데, POST 방식이나 세션을 사용하지 않은 이유는 무엇인가요?**

GET 파라미터가 가장 단순하고 RESTful합니다. 학년/반 선택은 서버 상태를 변경하지 않는 조회(Read) 작업이므로 GET이 적합합니다. 또한 URL로 직접 접근할 수 있어서 즐겨찾기나 공유가 쉽습니다. POST 방식은 폼 전송에 적합하고, 세션은 불필요한 상태를 유지하게 되어 서버 메모리를 낭비합니다.

## NEIS API / 데이터 연동

**16. NEIS(나이스) API는 어떤 API인가요? 인증은 어떻게 하나요?**

NEIS API는 교육행정정보시스템(나이스)에서 제공하는 공공 데이터 API로, 급식, 학사일정, 학교 기본 정보 등을 제공합니다. 공공데이터포털(data.go.kr)에서 API 키를 발급받아 요청 파라미터에 KEY 값으로 포함시키면 됩니다. RESTful 방식이며 JSON 또는 XML 형식으로 응답을 받을 수 있습니다.

**17. 급식 정보를 가져올 때 어떤 파라미터가 필요하고, 응답 데이터 구조는 어떻게 되나요?**

필요한 파라미터: 교육청 코드(ATPT_OFCDC_SC_CODE), 학교 코드(SD_SCHUL_CODE), 날짜(MLSV_YMD), 급식 종류(MLSV_SC_CODE - 1:조식, 2:중식, 3:석식). 응답은 JSON 배열 형태로 오며, 각 항목에는 DDISH_NM(요리 이름), CAL_INFO(칼로리), NTR_INFO(영양 정보) 등이 포함됩니다. 우리 프로젝트에서는 MealServiceResponse DTO로 매핑해서 사용합니다.

**18. 학사일정 데이터는 연 단위로 가져오는데, 특정 날짜만 필터링하는 로직은 어디서 처리하나요?**

NeisService에서 해당 연도의 전체 학사일정을 가져온 후, Java 스트림으로 현재 월(MonthDay)과 일치하는 항목만 필터링합니다. 전체 데이터를 캐싱해두고 월별로 필터링해서 보여주는 방식이라, API 호출을 월별로 따로 하지 않아도 됩니다.

**19. API 호출이 실패하면(네트워크 오류, 서버 다운 등) 사용자 화면에 어떻게 표시되나요?**

각 API 호출을 try-catch로 감싸서 예외 발생 시 빈 리스트를 반환합니다. 템플릿에서는 리스트가 비어있으면 '급식 정보가 없습니다' 또는 '시간표 정보가 없습니다' 같은 메시지를 표시합니다. 사용자는 일부 데이터가 누락된 상태로 페이지를 볼 수 있지만, 서버 자체는 다운되지 않고 정상 동작합니다.

**20. NEIS API의 하루 호출 제한은 어떻게 되나요? 제한을 초과하면 어떻게 대응하나요?**

공공데이터포털은 API마다 다르지만 일반적으로 일 1,000~10,000회 제한이 있습니다. 현재는 캐싱을 통해 같은 요청이 반복될 경우 API를 다시 호출하지 않도록 하고 있습니다. 또한 학교 하나만 대상이기 때문에 호출량이 많지 않습니다. 추후 사용자가 늘어나면 캐시 만료 전략을 더 정교하게 조정할 필요가 있습니다.

## 캐싱 / 성능

**21. 시간표 데이터를 ConcurrentHashMap에 캐싱하고 있는데, 캐시 만료 시간이나 크기 제한이 없는 이유는 무엇인가요?**

현재는 간단한 구현으로 시작했고, 실제 사용자가 많지 않아 메모리 이슈가 발생하지 않았습니다. 캐시는 서버 재시작 시 초기화되고, 시간표는 학기 중에는 자주 바뀌지 않는 데이터라 영구 캐싱도 나쁘지 않습니다. 추후 사용자가 늘어나면 ScheduledExecutor나 Caffeine 캐시를 도입해서 만료 정책을 추가할 계획입니다.

**22. ConcurrentHashMap을 사용한 이유는 무엇인가요? 일반 HashMap 또는 synchronized Map과 어떤 차이가 있나요?**

스프링의 기본 빈 스코프는 싱글톤이라 여러 요청이 동시에 같은 Service 인스턴스에 접근합니다. 일반 HashMap은 동시성 제어가 안 되어 데이터 손상(Corruption) 위험이 있습니다. synchronizedMap은 전체 맵에 락을 걸어 성능이 나쁩니다. ConcurrentHashMap은 세그먼트 단위 락킹으로 동시读写 성능이 좋아서 선택했습니다.

**23. 싱글 스레드 환경(톰캣)이면 그냥 HashMap을 써도 되는 거 아닌가요? 동시성 이슈에 대해 설명해주세요.**

톰캣은 기본적으로 멀티 스레드로 동작합니다. HTTP 요청 하나당 하나의 스레드를 할당해서 동시 요청을 병렬로 처리합니다. 즉 싱글 스레드 환경이 아닙니다. 따라서 동시에 HashMap에 접근하면 put과 get이 겹쳐서 무한 루프나 데이터 손실 같은 심각한 문제가 발생할 수 있습니다.

**24. Redis나 Ehcache 같은 전문 캐시 솔루션을 사용하지 않은 이유는 무엇인가요?**

현재 프로젝트 규모에서는 Redis 서버를 따로 운영하는 것이 오버스펙이라고 판단했습니다. ConcurrentHashMap 하나로 충분히 커버되는 수준의 데이터와 트래픽입니다. 또한 외부 의존성을 최소화해서 서버 실행 환경을 단순하게 유지하고 싶었습니다. 추후 분산 서버 환경이 필요해지면 그때 Redis를 도입할 예정입니다.

**25. 캐시가 쌓이면 메모리 부족 문제가 생기지 않을까요? 이에 대한 대비가 되어 있나요?**

현재 캐시되는 데이터는 근명고등학교 하나의 급식(1년치 학사일정, 월별 급식)과 시간표 데이터라서 수 KB ~ 수십 KB에 불과합니다. 메모리 부족이 발생할 가능성은 거의 없습니다. 만약 여러 학교를 추가하게 되면 캐시 크기가 커질 수 있으므로, 그 시점에 LRU(Least Recently Used) 기반의 Caffeine 캐시 또는 ScheduledEviction을 도입할 계획입니다.

## Thymeleaf / 프론트엔드

**26. Thymeleaf를 선택한 이유는 무엇인가요? JSP, FreeMarker, React 등과 비교했을 때 장단점은?**

Thymeleaf는 내추럴 템플릿(서버 없이도 브라우저에서 바로 열어볼 수 있음)이라는 장점이 있고, Spring Boot와 공식 통합이 잘 되어 있어 추가 설정이 거의 필요 없습니다. JSP는 Servlet API에 의존적이고 스프링 부트에서 권장하지 않습니다. React는 클라이언트 사이드 렌더링으로 학습 곡선이 높고, 단순한 정보 조회 페이지에는 오버스펙입니다.

**27. school.html에서 th:each, th:text 등의 문법을 사용하고 있는데, 서버사이드 렌더링의 장점은 무엇인가요?**

서버사이드 렌더링의 장점은: 1) 검색 엔진(SEO)이 HTML을 그대로 읽을 수 있습니다. 2) 초기 로딩 속도가 빠릅니다(클라이언트가 JS를 다운로드/실행할 필요 없음). 3) 자바스크립트가 꺼져 있어도 동작합니다. 4) 서버에서 직접 데이터를 가공해서 HTML로 내려주므로 클라이언트-서버 간 추가 API 통신이 필요 없습니다.

**28. CSS 프레임워크(Bootstrap, Tailwind 등)를 사용하지 않고 순수 CSS로만 만든 이유는 무엇인가요?**

프로젝트 규모가 작고 페이지 수가 적어서 CSS 프레임워크 없이도 충분히 커버 가능했습니다. 순수 CSS로 작성하면 외부 의존성이 없어 유지보수가 단순하고, 불필요한 클래스 없이 깔끔하게 작성할 수 있습니다. 또한 CSS를 직접 작성하면서 그라데이션 배경, 카드 레이아웃, 애니메이션 등 원하는 스타일을 자유롭게 구현할 수 있었습니다.

**29. 화면이 모바일에서도 잘 보이나요? 반응형 디자인을 고려했나요?**

아직 모바일 전용 최적화는 되어 있지 않습니다. 현재는 미디어 쿼리 없이 데스크탑 기준으로 작성되어 있어서, 모바일에서 보면 레이아웃이 깨질 수 있습니다. 추후 반응형 디자인을 적용해서 태블릿과 스마트폰에서도 편하게 볼 수 있도록 개선할 계획입니다.

**30. 타임리프의 레이아웃 기능(fragment, layout dialect)을 사용해보셨나요?**

이 프로젝트에서는 사용하지 않았습니다. 단일 페이지라서 레이아웃 기능이 필요하지 않았습니다. 하지만 페이지가 더 생긴다면, th:fragment와 th:replace를 사용해서 헤더/푸터 등 공통 영역을 모듈화하고, layout dialect로 공통 레이아웃 템플릿을 적용할 수 있습니다. 예를 들어 home.html과 school.html이 같은 네비게이션 바를 공유한다면 fragment로 분리하는 게 좋습니다.

## 데이터베이스

**31. 이 프로젝트는 MySQL 데이터베이스를 사용하는데, 학교 정보(급식, 시간표)는 DB에 저장하지 않고 API에서 바로 가져오는 이유는 무엇인가요?**

급식과 시간표는 NEIS에서 실시간으로 조회해야 최신 데이터를 보여줄 수 있습니다. DB에 저장하면 동기화 문제가 발생하고, 데이터 최신성 보장을 위해 결국 API를 다시 호출해야 합니다. 또한 학교 데이터는 외부 API에서 가져오는 것이므로 DB에 중복 저장하는 것은 불필요한 저장 공간 낭비입니다. 게시판 데이터(회원, 글)만 DB에 저장합니다.

**32. DB 스키마에서 사용자 테이블과 학교 정보 테이블의 관계를 설명해주세요.**

현재 DB에는 학교 정보 테이블이 따로 없습니다. 급식, 시간표, 학사일정은 모두 API에서 실시간으로 가져오고, 학교 코드와 교육청 코드는 설정 파일에 저장되어 있습니다. 사용자 테이블(아직 구현되지 않음)이 추가된다면, 사용자와 학교는 N:1 관계가 될 것입니다(한 학교에 여러 사용자).

**33. 한글 검색 기능은 어떻게 구현되어 있나요? MySQL에서 한글 검색의 어려운 점은 무엇인가요?**

SQL의 LIKE 연산자를 사용한 기본 검색입니다. MySQL에서 한글 검색의 어려운 점은 초성 검색이나 자음/모음 분리 검색이 기본적으로 지원되지 않는다는 점입니다. 예를 들어 'ㄱ'으로 시작하는 단어를 찾으려면 별도의 파싱 로직이 필요합니다. 현재는 LIKE '%키워드%' 형태의 단순 검색만 지원합니다.

**34. JPA를 사용했나요, 아니면 MyBatis나 JDBC를 사용했나요? 그 이유는?**

Spring Data JPA를 사용했습니다. JPA는 객체지향적인 데이터 접근이 가능하고, 메서드 이름만으로 쿼리를 자동 생성해주며, DB 벤더에 독립적입니다. 애노테이션 기반 매핑이 직관적이고, JPQL로 복잡한 쿼리도 작성 가능합니다. MyBatis는 SQL을 직접 작성해야 해서 유지보수가 번거롭고, JDBC는 너무 저수준입니다.

**35. Connection Pool은 어떻게 관리하나요? HikariCP에 대해 설명해주세요.**

Spring Boot는 기본적으로 HikariCP를 커넥션 풀로 사용합니다. HikariCP는 zero-overhead, 최소한의 락킹, 최적화된 알고리즘으로 가장 빠른 성능을 자랑합니다. 기본 설정으로 최대 10개의 커넥션을 유지하고, 필요에 따라 maxActive, maxIdle, connectionTimeout 등을 application.properties에서 조정할 수 있습니다.

## 배포 / DevOps

**36. 이 서버는 어디에 배포되어 있나요?**

학교 내부 서버에 배포되어 있습니다. 구체적인 위치는 학교 서버실 또는 특별실에 있는 Linux 서버(또는 개인 노트북/데스크탑)에서 실행 중입니다. 학교 내부망을 통해 접속하거나, 외부에서 접속할 수 있도록 포트 포워딩 설정이 되어 있을 수 있습니다.

**37. 배포 과정을 설명해주세요.**

1. 로컬에서 mvn clean package로 jar 파일 빌드
2. scp 또는 rsync로 jar 파일을 서버에 전송
3. 기존 프로세스 종료 (kill)
4. nohup java -jar starlog.jar & 로 백그라운드 실행
5. 로그 파일(tail -f)로 정상 기동 확인

이 과정이 번거롭다면, 추후 GitHub Actions로 CI/CD를 구축해서 push만 하면 자동 배포되도록 개선할 수 있습니다.

**38. 서버가 갑자기 다운되면 어떻게 대응하나요? 모니터링은 어떻게 하나요?**

현재는 별도의 모니터링 도구를 사용하지 않고, 문제가 생기면 개발자가 직접 SSH로 접속해서 확인하고 재시작하는 수동 대응 방식을 사용하고 있습니다. 추후 Spring Boot Actuator로 헬스 체크 엔드포인트를 추가하고, 우분투의 systemd 서비스로 등록해서 자동 재시작되도록 개선할 계획입니다.

**39. Linux 서버에서 Java 애플리케이션을 백그라운드로 실행할 때 어떤 명령어를 사용하나요?**

주로 nohup 명령어를 사용합니다: nohup java -jar starlog.jar > app.log 2>&1 &. nohup은 로그아웃 후에도 프로세스가 종료되지 않도록 해주고, &는 백그라운드 실행을 의미합니다. 더 안정적인 방법은 systemd 서비스로 등록하는 것입니다. /etc/systemd/system/starlog.service 파일을 만들어서 systemctl start starlog로 관리할 수 있습니다.

**40. HTTPS는 적용되어 있나요? SSL 인증서는 어떻게 발급받았나요?**

현재 내부망에서만 사용 중이라면 HTTP로도 충분할 수 있습니다. 외부에 공개한다면 Let's Encrypt에서 무료 SSL 인증서를 발급받아 적용할 계획입니다. certbot 도구를 사용하면 발급과 자동 갱신이 매우 간편합니다. Spring Boot에 SSL을 적용하려면 keystore를 생성하고 application.properties에 server.ssl 설정을 추가하면 됩니다.

## 보안

**41. API 키가 application.properties에 평문으로 저장되어 있는데, GitHub에 올라가면 어떻게 하나요?**

현재 개발 단계에서는 기본 키(데모용 또는 제한된 키)를 properties에 포함시켰습니다. 실제 운영 환경에서는: 1) application.properties를 .gitignore에 등록 2) OS 환경변수로 설정 (export NEIS_API_KEY=...) 3) application.properties에서 ${NEIS_API_KEY}로 참조. 이렇게 하면 키 값이 저장소에 노출되지 않습니다. 또는 스프링의 외부 설정 기능을 활용해 별도 프로필(application-prod.properties)로 분리할 수도 있습니다.

**42. SQL Injection 같은 공격은 어떻게 방어하나요?**

JPA(Spring Data JPA)는 기본적으로 파라미터 바인딩(PreparedStatement)을 사용하므로 SQL Injection에 안전합니다. 사용자 입력을 쿼리 문자열에 직접拼接하지 않고, :parameter 또는 ? placeholder를 사용해서 바인딩합니다. 또한 Thymeleaf는 HTML을 자동으로 이스케이프 처리해서 XSS 공격도 방어해줍니다.

**43. 사용자 입력값(grade, classNum)에 대한 검증은 어디서 하나요?**

Controller에서 @RequestParam으로 받은 후 서비스 호출 전에 검증합니다. grade는 1~3 사이 정수인지, classNum은 1~반개수 사이 정수인지 확인합니다. 현재는 간단한 조건문으로 검증하고 있지만, 추후 @Valid나 @Min/@Max 같은 Bean Validation 애노테이션을 DTO에 적용해서 더 체계적으로 검증할 수 있습니다.

**44. 서버 포트는 443(HTTPS)이나 8080 중 어떤 것을 사용하나요? 보안을 위해 방화벽 설정은 어떻게 되어 있나요?**

현재는 8080 포트를 사용하고 있습니다. 내부망에서만 사용 가능하도록 방화벽에서 특정 IP 대역만 허용하거나, iptables/ufw로 8080 포트 접근을 제한할 수 있습니다. 외부에 공개할 때는 80(HTTP)과 443(HTTPS) 포트를 열고, 8080은 내부에서만 접근 가능하게 방화벽을 설정할 예정입니다.

## 학교 특화 질문

**45. 근명고등학교 외에 다른 학교도 추가할 수 있나요? 얼마나 어렵나요?**

매우 쉽습니다. NEIS API는 모든 학교의 데이터를 제공하므로, 학교 코드(SD_SCHUL_CODE)와 교육청 코드(ATPT_OFCDC_SC_CODE)만 바꾸면 됩니다. 현재 application.properties에 설정된 이 값들만 다른 학교 코드로 변경하거나, 사용자가 학교를 선택할 수 있는 UI를 추가하면 됩니다.

**46. 학교 서버라는 게 구체적으로 어디에 있나요?**

학교 내부에 있는 Linux 서버에서 실행 중입니다. 서버실에 있는 전용 서버이거나, 특별실에 있는 컴퓨터일 수 있습니다. 학교 네트워크 환경에 따라 내부 IP로만 접속되거나, 포트 포워딩을 통해 외부에서도 접속 가능합니다.

**47. 선생님이나 학생들이 실제로 사용하고 있나요? 사용자 반응은 어떤가요?**

아직 공식적으로 배포되지 않았고, 개발자 본인이 테스트 중입니다. 정식 배포 후에는 교사와 학생들에게 안내하고 피드백을 받을 계획입니다. 학교 내에서 급식, 시간표, 학사일정을 한 번에 볼 수 있는 페이지가 없었기 때문에 반응이 좋을 것으로 기대합니다.

**48. 시간표에 나오는 과목들(공국, 국어, 수학 등) 중에 '공국'은 어떤 과목인가요?**

'공국'은 '공군 국가관'의 약자로, 공군 관련 인성 교육 과목입니다. 근명고등학교는 공군 특성화 학교(공군항공과학고등학교 등)일 가능성이 있어 일반 고등학교와 다른 특성화 과목이 시간표에 포함되어 있습니다.

**49. 이 프로젝트를 학교에서 어떻게 활용할 계획인가요?**

학교 홈페이지나 교내 정보 시스템에 통합하거나, 학생들이 자주 이용하는 포털 페이지를 제공할 계획입니다. 급식 정보(오늘 점심 메뉴), 시간표(매시간 확인), 학사일정(시험, 행사 확인)을 한 곳에서 편하게 볼 수 있게 하는 것이 목표입니다. 추가로 학교 커뮤니티 기능(익명 게시판 등)도 확장할 수 있습니다.

## 프로젝트 일반

**50. 이 프로젝트를 시작하게 된 계기는 무엇인가요?**

학교에서 급식, 시간표, 학사일정을 각각 다른 사이트에서 확인해야 하는 불편함이 있었습니다. 학교 홈페이지는 정보가 분산되어 있고 접근성이 좋지 않아서, 모든 정보를 한 페이지에서 볼 수 있는 서비스를 직접 만들면 좋겠다고 생각했습니다. 또한 Java와 Spring Boot를 실제 프로젝트에 적용해보고 싶었습니다.

**51. 개발하면서 가장 어려웠던 점은 무엇인가요? 어떻게 해결했나요?**

시간표 데이터를 가져오는 것이 가장 어려웠습니다. 처음에는 학교에서 사용하는 종합시간표 프로그램(comcigan)의 데이터를 파싱하려고 했는데, 이 방식은 불안정하고 유지보수가 어려웠습니다. 결국 NEIS API의 학급 시간표 조회 서비스(hisTimetable)로 전환해서 안정적으로 데이터를 가져올 수 있었습니다.

**52. 혼자서 개발했나요, 아니면 팀 프로젝트인가요?**

현재는 혼자서 개발하고 있습니다. 개인 프로젝트로 시작했고, 필요한 기능을 하나씩 추가하면서 학습과 개발을 병행하고 있습니다. 추후에 팀원을 모집해서 프로젝트를 확장할 가능성도 있습니다.

**53. 전체 개발 기간은 얼마나 걸렸나요?**

핵심 기능(급식, 시간표, 학사일정) 구현까지 약 2~3주 정도 소요되었습니다. 그 외 문서 정리, 프로젝트 구조 개선, 배포 설정 등 주변 작업을 포함하면 지속적으로 발전시키고 있는 프로젝트입니다.

**54. 앞으로 추가하고 싶은 기능은 무엇인가요?**

익명 게시판(학교 커뮤니티), 오늘의 명언/한 줄 글귀, 별자리 기반 랜덤 채팅(친구 찾기), 학교별 즐겨찾기, 모바일 반응형 디자인, 다크 모드, PWA(모바일 앱처럼 설치 가능) 등을 고려하고 있습니다.

**55. 이 프로젝트를 통해 무엇을 배웠나요?**

Spring Boot를 실제 프로젝트에 적용하는 전체 과정을 경험했습니다. 특히 외부 API 연동(RestTemplate, JSON 파싱), 서버 사이드 렌더링(Thymeleaf), 캐싱 전략(ConcurrentHashMap), 그리고 실서버 배포(nohup, 프록시 설정)까지 실전 경험을 쌓을 수 있었습니다. 또한 REST API 설계와 레이어드 아키텍처의 중요성을 체감했습니다.

**56. STARLOG라는 이름은 어떻게 지었나요? 의미가 있나요?**

STARLOG는 'Star'와 'Log'의 합성어입니다. 별처럼 반짝이는 학교 생활의 기록(Log)이라는 의미를 담고 있습니다. 학교에서의 매일매일(급식, 시간표, 일정)을 별빛처럼 기록하고 보여주는 서비스라는 컨셉으로 지었습니다.

**57. 같은 주제로 다시 만든다면 무엇을 다르게 하고 싶나요?**

처음부터 NEIS API만 사용할 것입니다. 처음에는 comcigan-parser로 시간표를 파싱하려고 했다가 문제가 많아서 NEIS API로 갈아탔습니다. 또한 초기부터 CI/CD를 구축하고, 캐시 전략(만료 시간 포함)을 더 체계적으로 설계했을 것입니다. 그리고 테스트 코드를 먼저 작성하는 TDD 방식으로 개발할 것입니다.

## 심화 질문

**58. Java의 리플렉션(Reflection)이 무엇이고, 스프링 내부에서 어떻게 사용되나요?**

리플렉션은 실행 중인 클래스의 정보(메서드, 필드, 생성자)를 동적으로 분석하고 조작하는 기능입니다. 스프링은 리플렉션을 광범위하게 사용합니다: @Autowired 필드 주입, @ComponentScan으로 클래스 찾기, 프록시 생성(AOP), JPA 엔티티 매핑 등이 모두 리플렉션을 기반으로 동작합니다. 덕분에 우리는 new 키워드 없이도 객체를 주입받아 사용할 수 있습니다.

**59. AOP(Aspect Oriented Programming)가 무엇인가요? 이 프로젝트에 적용할 수 있는 부분이 있을까요?**

AOP는 핵심 로직과 공통 관심사(로깅, 트랜잭션, 보안)를 분리하는 프로그래밍 패러다임입니다. 이 프로젝트에 적용할 수 있는 부분: 1) 모든 API 호출의 시작/종료 시간 로깅 2) 캐시 히트율 모니터링 3) API 호출 실패 시 재시도 로직. @Around 어노테이션으로 메서드 실행 전후에 공통 로직을 삽입할 수 있습니다.

**60. 멀티스레드 환경에서 동시성 제어를 어떻게 하나요?**

Java는 다양한 동시성 제어 도구를 제공합니다: synchronized 키워드(메서드나 블록 레벨 락), volatile(가시성 보장), ReentrantLock(명시적 락), ConcurrentHashMap(세그먼트 단위 동시성), AtomicInteger/CAS(락프리 동기화). 우리 프로젝트에서는 ConcurrentHashMap을 사용해서 캐시의 스레드 안전성을 확보했습니다.

**61. JVM의 메모리 구조에 대해 설명해주세요. 캐시 데이터는 어디에 저장되나요?**

JVM 메모리는 크게 Heap(모든 객체가 저장되는 공간), Stack(메서드 호출과 지역 변수), Method Area(클래스 정보, static 변수), PC Register, Native Method Stack으로 나뉩니다. 우리의 ConcurrentHashMap 캐시 데이터는 Heap 영역 중 Young Generation(Eden/Survivor)에서 시작해서, 오래 살아남으면 Old Generation으로 이동합니다.

**62. 가비지 컬렉션은 어떻게 동작하나요? 캐시 데이터가 GC 대상이 되지 않게 하려면?**

GC는 JVM이 더 이상 참조되지 않는 객체를 자동으로 메모리에서 해제하는 과정입니다. Young Generation에서는 Minor GC(복사 방식), Old Generation에서는 Major GC(마크-스윕-컴팩트)가 발생합니다. 캐시 데이터가 GC 대상이 되지 않게 하려면: 1) ConcurrentHashMap 인스턴스를 static이나 빈으로 유지해서 Strong Reference를 계속 참조 2) WeakHashMap 대신 사용 (WeakReference는 GC 대상이 될 수 있음). 우리 프로젝트의 캐시는 빈으로 등록되어 있어 GC 대상이 되지 않습니다.

**63. RESTful API의 원칙에 대해 설명해주세요. 이 프로젝트의 API 설계가 RESTful한가요?**

RESTful API의 핵심 원칙: 1) 자원(Resource) 기반 URL (/school, /meal) 2) HTTP 메서드로 행위 표현 (GET 조회, POST 생성, PUT 수정, DELETE 삭제) 3) 무상태(Stateless) - 각 요청이 독립적 4) JSON/XML로 표현. 우리 프로젝트의 /school?grade=&classNum= 는 RESTful 원칙을 따릅니다. 자원(school 정보)을 GET으로 조회하고, 상태를 서버에 저장하지 않습니다.

**64. HTTP 메서드의 차이와 언제 사용해야 하는지 설명해주세요.**

GET: 데이터 조회 (서버 상태 변경 없음, 캐시 가능) POST: 새로운 리소스 생성 (멱등성 없음) PUT: 리소스 전체 교체 (멱등성 있음) PATCH: 리소스 일부 수정 DELETE: 리소스 삭제. 우리 프로젝트에서는 모든 요청이 조회이므로 GET만 사용하고 있습니다. 추후 게시판 기능을 추가하면 POST로 글 쓰기, DELETE로 글 삭제를 구현할 수 있습니다.

**65. 쿠키와 세션의 차이점은 무엇인가요? 이 프로젝트에서 세션을 사용하나요?**

쿠키는 클라이언트(브라우저)에 저장되는 작은 데이터(최대 4KB)이고, 세션은 서버 메모리에 저장되는 데이터입니다. 쿠키는 자동으로 요청마다 전송되고, 세션은 쿠키에 저장된 세션 ID로 식별합니다. 현재 이 프로젝트는 로그인 기능이 없어서 세션을 사용하지 않습니다. 추후 인증 기능을 추가하면 HttpSession이나 Spring Session을 사용할 수 있습니다.

**66. OSI 7계층에 대해 간단히 설명해주세요. HTTP는 몇 계층 프로토콜인가요?**

OSI 7계층: 1계층 물리(Physical) - 케이블, 신호 2계층 데이터 링크(Data Link) - MAC 주소, 이더넷 3계층 네트워크(Network) - IP, 라우팅 4계층 전송(Transport) - TCP/UDP, 포트 5계층 세션(Session) - 연결 관리 6계층 표현(Presentation) - 암호화, 인코딩 7계층 응용(Application) - HTTP, FTP, SMTP. HTTP는 7계층(응용 계층) 프로토콜입니다.

**67. TCP와 UDP의 차이는 무엇인가요? HTTP는 어떤 것을 사용하나요?**

TCP는 연결 지향적(3-way handshake), 데이터 전송 보장(재전송, 순서 보장), 느리지만 신뢰성이 높습니다. UDP는 비연결형, 데이터 전송 미보장, 빠르지만 신뢰성이 낮습니다(실시간 스트리밍, 게임에 사용). HTTP는 TCP를 기반으로 합니다. 웹 페이지는 데이터가 손실되면 안 되기 때문에 신뢰성 있는 TCP를 사용하는 것이 적합합니다.
